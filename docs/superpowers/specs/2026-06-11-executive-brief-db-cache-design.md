# AI Weekly Executive Brief ŌĆö DB-backed daily cache with force-regenerate

**Date:** 2026-06-11
**Status:** Approved
**Area:** `backend/` (FastAPI + Tortoise ORM + PostgreSQL), `frontend/` (React + React Query)

## Problem

The executive page renders an "AI Weekly Executive Brief" generated by an LLM. Today the
brief is produced **on demand** inside `get_executive_summary()` and cached only in
**process memory** (`_weekly_brief_cache`, 60-minute TTL) in
`backend/app/services/analytics.py`. Consequences:

- A cache miss makes the page request **block on a paid LLM call**.
- The cache is lost on every restart/deploy and is not shared across worker processes.
- There is no way to deliberately refresh the brief.

## Goals

1. Regenerate the brief on a **daily** schedule.
2. Persist the brief in the **database** instead of in process memory.
3. Provide a **force-regenerate** button (admin only).
4. The page GET must **never block** on the LLM.

Non-goals: changing the prompt, KPIs, or trend computations; multi-language briefs;
historical brief browsing UI.

## Decisions (confirmed with user)

- **Regenerate auth:** admin only (`Depends(require_admin)`), matching `agencies.py`.
- **Cold start:** GET never blocks ŌĆö it reads the latest stored brief; if none exists yet
  it returns placeholder text. The scheduler populates it shortly after startup.
- **Schedule:** every 24h via APScheduler `IntervalTrigger`, mirroring the existing
  `agency_chat_test` job (kick once at startup + recurring).

## Design

### 1. Data model

New Tortoise model `ExecutiveBrief` in `backend/app/models/executive_brief.py`, registered
in `backend/app/models/__init__.py`, with an Aerich migration.

| field | type | notes |
|---|---|---|
| `id` | uuid PK | consistent with other models |
| `content` | text | generated Thai brief |
| `status` | varchar(16) | `"ok"` or `"error"` |
| `generated_at` | timestamptz | indexed; default now |

Append-only: each regeneration **inserts** a row; reads take the latest by
`generated_at` (`order_by("-generated_at").first()`). No upsert; free audit trail.

Only the brief is cached. KPIs and `monthlyTrend` remain **computed live** on each GET
(cheap DB aggregates).

### 2. Service refactor ŌĆö `backend/app/services/analytics.py`

- Extract `_compute_executive_metrics()` ŌåÆ returns the stats dict + `monthlyTrend`
  (used by both the GET path and the regen path; removes duplication).
- Extract `_build_brief_prompt(metrics)` ŌåÆ the Thai prompt currently inline at
  lines ~239-249.
- New `regenerate_weekly_brief() -> ExecutiveBrief`: compute metrics ŌåÆ build prompt ŌåÆ
  call LLM ŌåÆ **insert an `ExecutiveBrief` row** (`status="error"` + fallback text on
  failure) ŌåÆ return it. Single source of truth for both scheduler and button.
- `get_executive_summary()`: compute KPIs/trend live as today, but set `weeklyBrief`
  from the **latest `ExecutiveBrief` row** (placeholder text if the table is empty).
  No LLM call on this path.
- **Remove** in-memory globals `_weekly_brief_cache`, `_weekly_brief_cache_time`,
  `_weekly_brief_lock`, and the LLM call inside the old `_get_weekly_brief`.

### 3. Config ŌĆö `backend/app/config.py`

- **Remove** `WEEKLY_BRIEF_CACHE_TTL_MINUTES` (and from the masked/exported groups).
- **Add** `BRIEF_REGEN_INTERVAL_HOURS: int = 24`.
- Keep `WEEKLY_BRIEF_TIMEOUT`.

### 4. Scheduler ŌĆö `backend/app/scheduler.py`

Add a `regenerate_brief_job` that calls `regenerate_weekly_brief()`. In
`start_scheduler()`: `asyncio.create_task(regenerate_weekly_brief())` once at startup
(background ŌĆö does not block app start) and
`scheduler.add_job(..., IntervalTrigger(hours=settings.BRIEF_REGEN_INTERVAL_HOURS))`.
Errors are caught/logged like `agency_chat_item`.

### 5. Endpoint ŌĆö `backend/app/routers/executive_summary.py`

`POST /executive-summary/regenerate`, `Depends(require_admin)`. Calls
`regenerate_weekly_brief()` and returns `{ weeklyBrief, generatedAt, status }`.

### 6. Frontend ŌĆö `frontend/src/features/executive/`

- `executiveApi.ts`: add `regenerateExecutiveSummary()` (POST).
- `useExecutive.ts`: add `useRegenerateExecutiveSummary()` mutation; on success
  `invalidateQueries(['executive','summary'])`.
- Brief card: add a "Regenerate" button with a loading spinner, rendered only when
  `useAuth().isAdmin` is true.

## Testing (TDD ŌĆö write failing tests first)

Backend (`backend/tests/services/`, `backend/tests/`):

1. `regenerate_weekly_brief()` inserts an `ExecutiveBrief` row and returns it (LLM mocked).
2. On LLM failure, a row is still inserted with `status="error"` and fallback content.
3. `get_executive_summary()` returns the **latest** brief content and makes **no** LLM call.
4. `get_executive_summary()` returns placeholder text when the table is empty.
5. `POST /executive-summary/regenerate` returns 401/403 without admin; 200 with admin.

Frontend: mutation triggers POST and invalidates the summary query (existing test style).

## Risks / notes

- Backend GET remains public (pre-existing); only the new regenerate endpoint is gated.
  Out of scope to change GET auth here.
- Removing `WEEKLY_BRIEF_CACHE_TTL_MINUTES` is a breaking config key removal ŌĆö update
  `.env.example` if it references it.
