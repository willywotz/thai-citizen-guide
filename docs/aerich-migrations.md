# Database Migrations with aerich — the correct way

This project uses [aerich](https://github.com/tortoise/aerich) (Tortoise ORM migrations). Migrations live in `backend/migrations/models/` and are numbered `N_<timestamp>_<name>.py`.

**Read this before creating, editing, or hand-writing any migration.** Most of our past migration breakage came from ignoring the single rule below.

## The one rule that matters: never fake `MODELS_STATE`

Every migration file ends with a `MODELS_STATE = (...)` base64 blob. This is **aerich's snapshot of the models at that migration**, and it is the **diff baseline for the *next* `aerich migrate`**. It is NOT executed at upgrade time — only the `upgrade()`/`downgrade()` SQL runs.

If you hand-write a migration and **copy `MODELS_STATE` forward from the previous migration** (because "aerich wouldn't run"), the snapshot is now **stale**: it does not describe the schema your migration produced. The next time someone runs `aerich migrate`, aerich diffs the real models against that stale snapshot and emits **spurious, duplicated operations** — e.g. re-adding a column that already exists. That is the root cause of errors like:

```
tortoise.exceptions.OperationalError: column "message_id" of relation "connection_logs" already exists
```

**So: `MODELS_STATE` must always be generated by `aerich migrate` itself, run against a database that already has every prior migration applied.** Do not hand-author or carry it forward.

## Correct workflow to add a migration

Run from `backend/` (aerich reads `[tool.aerich]` in `backend/pyproject.toml`). aerich needs a reachable database at `DATABASE_URL` (default `postgres://postgres:postgres@localhost:5432/chatbot`; point it at the docker Postgres or a scratch DB).

```bash
cd backend

# 1. Make sure the target DB has ALL existing migrations applied first.
#    This is what makes the regenerated MODELS_STATE correct.
.venv/bin/aerich upgrade

# 2. Change the Tortoise model(s) in app/models/.

# 3. Generate the migration (regenerates MODELS_STATE correctly).
.venv/bin/aerich migrate --name <short_description>

# 4. Review the generated file (see checklist below), then apply + test.
.venv/bin/aerich upgrade
```

Useful commands: `aerich heads` (pending), `aerich history` (applied), `aerich downgrade -v -1` (revert one; add `-d` to delete the file), `aerich upgrade --fake` (mark applied without running SQL — for reconciling an already-correct schema).

## Review checklist after `aerich migrate`

1. **`git status` must show exactly ONE new file** under `backend/migrations/models/` and **nothing modified or deleted**. aerich has, in this repo, mis-numbered or deleted existing migration files (e.g. clobbering `9_..._embedding_vector_index.py`). If any existing migration changed, **STOP and `git checkout` / delete to restore it**, then investigate — never commit a run that rewrote history.
2. **Read the `upgrade()`/`downgrade()` SQL.** Confirm it matches your model change and nothing else.
3. **Hand-add raw-SQL objects aerich does not track:** `CREATE EXTENSION`, and custom indexes created via raw SQL (e.g. `USING hnsw ... vector_cosine_ops`, `USING gin ... gin_trgm_ops`). aerich only tracks column/table/plain-index changes from the model, so any extension or expression index we manage manually must be added to `upgrade()` (and reversed in `downgrade()`), idempotently: `CREATE EXTENSION IF NOT EXISTS ...`, `DROP INDEX IF EXISTS ...`.
4. **Confirm `MODELS_STATE` was regenerated** (it should differ from the previous migration's blob when the models changed).

## If aerich genuinely cannot run (last resort)

If there is no reachable DB and you must hand-write a migration:
- Write only the `upgrade()`/`downgrade()` SQL by hand (idempotent).
- **Do NOT invent or carry `MODELS_STATE`.** Leave a clear note in the commit body and in this file's owner that **`MODELS_STATE` is stale and MUST be regenerated by a real `aerich migrate` (against a DB with all migrations applied) before the next model-changing migration.** Until then, the executable SQL deploys fine, but nobody may run `aerich migrate` on top of it.

## Startup / "already exists" at boot

`app/database.py:init_db()` runs `aerich upgrade` **and then** `Tortoise.generate_schemas(safe=True)`. `generate_schemas` creates tables directly from the models **without recording anything in aerich's version table**. On a **reused/dirty DB volume** this desyncs aerich's history from the real schema, so the next boot's `aerich upgrade` re-applies old migrations and fails with `... already exists`.

- Locally: reset the volume — `docker compose down -v && docker compose up`. On a fresh DB, migrations `0 → head` apply cleanly.
- To keep existing data: `aerich upgrade --fake` to the current head to mark history consistent.
- Prod is unaffected: it has a consistent aerich history, so only new migrations run.

## Recovery (local/dev only — destroys migration history)

Per aerich docs, to reset corrupted migration history without touching the actual schema:
1. Drop the `aerich` table from the DB.
2. Delete the `migrations/models/` directory.
3. `aerich init-db`.

Never do this against dev/prod databases.

## Config reference

`backend/pyproject.toml`:
```toml
[tool.aerich]
tortoise_orm = "app.config.TORTOISE_ORM"
location = "./migrations"
src_folder = "./."
```
`TORTOISE_ORM` (in `app/config.py`) must keep `aerich.models` in its `apps.models.models` list — that is the table aerich uses to track applied versions.
