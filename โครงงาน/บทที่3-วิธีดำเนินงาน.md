# บทที่ 3 วิธีดำเนินงาน

บทนี้อธิบายกระบวนการและวิธีดำเนินงานในการพัฒนาระบบ AI Chatbot Portal สำหรับสังเคราะห์คำตอบจากหน่วยงานภาครัฐ ครอบคลุมตั้งแต่ขั้นตอนการวางแผน การออกแบบสถาปัตยกรรมระบบ การออกแบบฐานข้อมูล การกำหนด API Endpoints ไปจนถึงการออกแบบส่วนติดต่อผู้ใช้และกลไกการทำงานของ AI Multi-Agent Routing โดยทุกส่วนถูกออกแบบให้สอดคล้องกับวัตถุประสงค์ที่กำหนดไว้ในบทที่ 1 และอ้างอิงทฤษฎีที่นำเสนอในบทที่ 2

## 3.1 ขั้นตอนการดำเนินงาน

การพัฒนาระบบ AI Chatbot Portal ดำเนินงานตามขั้นตอนดังนี้:

1. **ศึกษาและวิเคราะห์ความต้องการ** — ศึกษาปัญหาการเข้าถึงข้อมูลภาครัฐ วิเคราะห์ความต้องการของผู้ใช้ทั้งประชาชนและเจ้าหน้าที่ กำหนด Functional Requirements และ Non-functional Requirements
2. **ออกแบบสถาปัตยกรรมระบบ** — ออกแบบ System Architecture, Database Schema, API Endpoints, UI/UX
3. **พัฒนา Backend API** — พัฒนา REST API ด้วย FastAPI, ระบบ Authentication, ระบบ Chat, MCP Server
4. **พัฒนา Frontend** — พัฒนา Public Portal และ Admin Portal ด้วย React + TypeScript
5. **พัฒนาระบบ AI Multi-Agent** — พัฒนา Keyword Detection, LangGraph Agent Routing, LLM Answer Synthesis
6. **จัดทำ Docker Configuration** — เขียน Dockerfile, docker-compose.yaml, Nginx config
7. **ทดสอบระบบ** — ทดสอบ Unit Test, Integration Test, UI Testing
8. **ปรับปรุงและสรุปผล** — แก้ไขข้อบกพร่อง จัดทำเอกสาร

### 3.2.1 Client Layer

Frontend พัฒนาด้วย React 18 + TypeScript ใช้ Vite เป็น Build Tool ให้ Hot Module Replacement (HMR) ระหว่างพัฒนา UI Component ใช้ shadcn/ui ที่สร้างบน Radix UI จัดการ Style ด้วย TailwindCSS ระบบ Routing ใช้ React Router v6 การเรียก API ใช้ Axios ร่วมกับ TanStack React Query สำหรับ caching และ state management

### 3.2.2 Infrastructure Layer

Nginx ทำหน้าที่เป็น Reverse Proxy รับ Request ทั้งหมดที่ Port 80 โดย Request ที่ขึ้นต้นด้วย `/api/` จะถูก Proxy ไปยัง Backend ที่ Port 8000 ส่วน Request อื่น ๆ จะ Serve Static Files ของ Frontend Build

### 3.2.3 Backend Layer

FastAPI Backend ให้บริการ 3 ช่องทาง:
- **REST API** (`/api/v1/*`): Endpoints สำหรับ Authentication, Chat, Agency CRUD, Dashboard, Feedback
- **MCP SSE** (`/sse`, `/messages/`): Server-Sent Events transport สำหรับ MCP Client
- **MCP Streamable-HTTP** (`/mcp/`): HTTP transport สำรอง

### 3.2.4 Data Layer

PostgreSQL 16 (Alpine) เป็นฐานข้อมูลหลัก เชื่อมต่อผ่าน Tortoise ORM ด้วย asyncpg driver ใช้ Aerich สำหรับ Database Migration

| Method | Endpoint | คำอธิบาย |
|--------|----------|----------|
| POST | /api/v1/auth/login | เข้าสู่ระบบ รับ JWT Token |
| POST | /api/v1/auth/signup | ลงทะเบียนผู้ใช้ใหม่ |

### 3.4.2 Chat API

| Method | Endpoint | คำอธิบาย |
|--------|----------|----------|
| POST | /api/v1/chat | ส่งคำถาม รับคำตอบสังเคราะห์จากหลายหน่วยงาน |

Request Body:
```json
{
  "query": "ยาพาราเซตามอลต้องขึ้นทะเบียนไหม",
  "conversation_id": "uuid (optional)"
}
```

Response:
```json
{
  "answer": "คำตอบที่สังเคราะห์แล้ว",
  "agencies": ["สำนักงาน อย."],
  "references": [{"title": "...", "url": "..."}],
  "agent_steps": [{"agent": "fda", "status": "completed", "duration": 1.2}],
  "confidence": 0.85,
  "conversation_id": "uuid"
}
```

## 3.6 การทำงานของ AI Multi-Agent Routing

ระบบ Chat ทำงานตามขั้นตอนดังนี้:

> **[รูปภาพ 3.9: Flowchart การทำงานของระบบ Chat]**
> แสดง Flow ดังนี้:
>
> ```
> [ผู้ใช้พิมพ์คำถาม]
>         │
>         ▼
> [1. Keyword Detection]
> ตรวจจับ keyword ภาษาไทยในคำถาม เช่น:
> - "ยา", "อาหาร", "เครื่องสำอาง" → อย.
> - "ภาษี", "ลดหย่อน", "สรรพากร" → กรมสรรพากร
> - "บัตรประชาชน", "ทะเบียนบ้าน" → กรมการปกครอง
> - "ที่ดิน", "โฉนด", "ราคาประเมิน" → กรมที่ดิน
>         │
>         ▼
> [2. ดึง Agency Config จาก DB]
> ดึง response_schema, data_scope, endpoint_url
> ของหน่วยงานที่ตรวจพบ
>         │
>         ▼
> [3. Query Agencies แบบ Parallel]
> ใช้ asyncio.gather() ส่ง query
> ไปยังทุกหน่วยงานที่ตรวจพบพร้อมกัน
>         │
>         ▼
> [4. สร้าง Schema-Guide Prompt]
> นำ response_schema ของแต่ละหน่วยงาน
> มาสร้าง prompt ชี้แนะ LLM
>         │
>         ▼
> [5. LLM Synthesize Answer]
> ส่ง prompt + ข้อมูลจากหน่วยงาน
> ไปยัง LLM (OpenThai GPT)
> เพื่อสังเคราะห์คำตอบเดียว
>         │
>         ▼
> [6. Return Response]
> ส่งกลับ: answer, agencies, references,
> agent_steps, confidence
>         │
>         ▼
> [7. บันทึกลง DB]
> สร้าง/อัปเดต Conversation + Message
> ```

### 3.6.1 Keyword-based Agency Detection

ระบบใช้ฟังก์ชัน `detect_agencies()` ตรวจจับ keyword ภาษาไทยในคำถามของผู้ใช้ โดยเปรียบเทียบกับรายการ keyword ที่กำหนดไว้สำหรับแต่ละหน่วยงาน หากไม่พบ keyword ที่ตรงกัน ระบบจะ fallback ไปยังหน่วยงานเริ่มต้น (อย.)

ตัวอย่าง Keyword Mapping:

| หน่วยงาน | Keywords |
|----------|----------|
| อย. (FDA) | ยา, อาหาร, เครื่องสำอาง, อย., พาราเซตามอล, นำเข้า, ผลิตภัณฑ์สุขภาพ |
| กรมสรรพากร | ภาษี, ลดหย่อน, สรรพากร, vat, ยื่นแบบ, เงินได้ |
| กรมการปกครอง | บัตรประชาชน, ทะเบียนราษฎร์, ทะเบียนบ้าน, ปกครอง, เปลี่ยนชื่อ, แจ้งเกิด |
| กรมที่ดิน | ที่ดิน, โฉนด, ราคาประเมิน, จดทะเบียน, รังวัด, โอนที่ดิน |

### 3.6.2 LangGraph Agent Routing

นอกจาก Keyword Detection แล้ว ระบบยังมีตัวเลือก LangGraph-based Routing ที่ใช้ LLM ตัดสินใจ route คำถามไปยังหน่วยงานที่เหมาะสม โดยพิจารณาจาก `data_scope` ของแต่ละหน่วยงานในฐานข้อมูล วิธีนี้มีความยืดหยุ่นสูงกว่า Keyword Detection เนื่องจากสามารถเข้าใจบริบทของคำถามได้ดีกว่า

### 3.6.3 Parallel Agency Querying

เมื่อตรวจพบหน่วยงานที่เกี่ยวข้องแล้ว ระบบจะใช้ `asyncio.gather()` ส่ง query ไปยังทุกหน่วยงานพร้อมกัน (Parallel) เพื่อลดเวลารอคอย แทนที่จะส่งทีละหน่วยงาน (Sequential)

### 3.6.4 LLM Answer Synthesis

ข้อมูลที่ได้จากแต่ละหน่วยงานจะถูกรวมเข้ากับ Schema-Guide Prompt แล้วส่งไปยัง LLM (OpenThai GPT ผ่าน AI Gateway ที่ `thaillm.or.th`) เพื่อสังเคราะห์เป็นคำตอบเดียวที่ครอบคลุมข้อมูลจากทุกหน่วยงาน

> [MCP Client (เช่น Claude Desktop)]
>         │
>         │ GET /sse (เปิด SSE Stream)
>         ▼
> [SSE Transport Layer]
>         │
>         │ POST /messages/ (JSON-RPC)
>         ▼
> [FastMCP Server]
>   ├── Resource: agencies://list
>   │   └── คืนข้อมูลหน่วยงานทั้งหมด (JSON)
>   └── Tool: list_agency
>       └── คืน {agencies: [...], total: N}
>         │
>         ▼
> [PostgreSQL — ตาราง agencies]
> ```

MCP Server ใช้ FastMCP Library สร้าง 2 ช่องทาง:
- **Resource** `agencies://list`: คืนรายการหน่วยงานที่ active ทั้งหมดในรูป JSON string
- **Tool** `list_agency`: คืน dictionary ที่มี key `agencies` (array) และ `total` (count)

ข้อมูลที่เปิดเผยต่อ MCP Client ได้แก่ id, name, description, connection_type, data_scope, endpoint_url, expected_payload

> ┌── docker-compose ─────────────────────────────────────┐
> │                                                       │
> │  ┌────────────┐  ┌──────────────┐  ┌──────────────┐  │

> │  │  Port: 80   │──│  Port: 8000  │──│  Port: 5432  │  │
> │  │  Dockerfile │  │  Dockerfile  │  │  postgres:16 │  │
> │  │  (Frontend  │  │  (FastAPI +  │  │  -alpine     │  │
> │  │   + Proxy)  │  │   MCP)       │  │              │  │
> │  └────────────┘  └──────────────┘  └──────────────┘  │
> │                                                       │
> │  Network: chatbot-network                             │

| Service | Image/Build | Port | หน้าที่ |
|---------|-------------|------|---------|
| postgres | postgres:16-alpine | 5432 | ฐานข้อมูล PostgreSQL พร้อม Volume สำหรับ Persistent Storage |
| backend | ./backend/Dockerfile | 8000 | FastAPI + MCP Server พร้อม Environment Variables สำหรับ DB, JWT, LLM |
| nginx | ./Dockerfile | 80 | Build Frontend (Vite) + Serve Static Files + Reverse Proxy ไปยัง Backend |
