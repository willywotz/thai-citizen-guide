# บทที่ 2 ทฤษฎีและงานวิจัยที่เกี่ยวข้อง

## 2.1 Large Language Model (LLM)

Large Language Model หรือ LLM คือโมเดลปัญญาประดิษฐ์ที่ถูกฝึกด้วยข้อมูลข้อความจำนวนมหาศาล โดยใช้สถาปัตยกรรม Transformer ซึ่งถูกนำเสนอครั้งแรกในงานวิจัย "Attention Is All You Need" โดย Vaswani et al. (2017) สถาปัตยกรรมนี้ใช้กลไก Self-Attention ที่ทำให้โมเดลสามารถเข้าใจความสัมพันธ์ของคำในประโยคได้อย่างมีประสิทธิภาพ

LLM สมัยใหม่ เช่น GPT-4, Claude, Gemini และ OpenThai GPT ถูกฝึกด้วยข้อมูลหลายร้อยพันล้านคำ สามารถทำงานได้หลากหลาย ทั้งการตอบคำถาม การสรุปความ การแปลภาษา และการสร้างข้อความ ในโครงงานนี้ใช้ LLM ผ่าน AI Gateway ของ thaillm.or.th เพื่อสังเคราะห์คำตอบจากข้อมูลของหลายหน่วยงานให้เป็นคำตอบเดียวที่ครอบคลุม

## 2.2 Multi-Agent System

Multi-Agent System (MAS) คือระบบที่ประกอบด้วย Agent อิสระหลายตัวทำงานร่วมกันเพื่อบรรลุเป้าหมายที่ซับซ้อน แต่ละ Agent มีความสามารถเฉพาะทางและสามารถสื่อสารกันได้ แนวคิดนี้มาจากสาขา Distributed Artificial Intelligence

ในบริบทของ LLM-based System นั้น Multi-Agent Architecture หมายถึงการใช้ LLM Agent หลายตัว โดยแต่ละตัวรับผิดชอบงานเฉพาะด้าน เช่น Agent สำหรับดึงข้อมูลจาก อย., Agent สำหรับกรมสรรพากร เป็นต้น แล้วมี Orchestrator Agent ทำหน้าที่ประสานงานและสังเคราะห์ผลลัพธ์

ข้อดีของ Multi-Agent System:
- แต่ละ Agent มีความเชี่ยวชาญเฉพาะด้าน ทำให้คำตอบมีคุณภาพสูง
- สามารถ query หลาย Agent พร้อมกัน (Parallel) ลดเวลาในการตอบ
- เพิ่มหรือลด Agent ได้อิสระโดยไม่กระทบระบบโดยรวม (Scalable)
- แต่ละ Agent สามารถใช้แหล่งข้อมูลที่แตกต่างกันได้

## 2.3 LangChain และ LangGraph

**LangChain** เป็น Framework สำหรับพัฒนาแอปพลิเคชันที่ใช้ LLM โดยมีโมดูลสำคัญ ได้แก่ Models (การเชื่อมต่อกับ LLM), Prompts (การจัดการ Prompt Template), Chains (การเชื่อมต่อหลาย ๆ ขั้นตอนเข้าด้วยกัน) และ Agents (การให้ LLM ตัดสินใจเลือกเครื่องมือที่จะใช้)

**LangGraph** เป็น Library ที่สร้างบน LangChain สำหรับสร้าง Multi-Agent Workflow ในรูปแบบ Graph โดยแต่ละ Node ใน Graph เป็น Agent หรือ Function และ Edge เป็นเส้นทางการไหลของข้อมูล LangGraph รองรับการทำงานแบบ Cyclic ทำให้สามารถสร้าง Workflow ที่ซับซ้อนได้ เช่น การวนลูปจนกว่าจะได้คำตอบที่น่าพอใจ

ในโครงงานนี้ LangGraph ถูกใช้เพื่อสร้าง Agent Routing Graph ที่ตัดสินใจว่าคำถามของผู้ใช้ควรส่งไปยัง Agent ของหน่วยงานใด โดยพิจารณาจาก data_scope ของแต่ละหน่วยงาน

## 2.4 Model Context Protocol (MCP)

Model Context Protocol (MCP) เป็นโปรโตคอลแบบเปิด (Open Protocol) ที่พัฒนาโดย Anthropic สำหรับการเชื่อมต่อ LLM กับแหล่งข้อมูลภายนอก MCP กำหนดมาตรฐานในการสื่อสารระหว่าง LLM Client กับ MCP Server โดย MCP Server ทำหน้าที่เปิดเผย Resources (ข้อมูล) และ Tools (ฟังก์ชัน) ให้ LLM เข้าถึงได้

โครงสร้างของ MCP ประกอบด้วย:
- **MCP Host**: แอปพลิเคชันที่ต้องการใช้ข้อมูล (เช่น Claude Desktop)
- **MCP Client**: ตัวกลางที่เชื่อมต่อกับ MCP Server
- **MCP Server**: ให้บริการข้อมูลและเครื่องมือ

ในโครงงานนี้ใช้ FastMCP Library สร้าง MCP Server ที่เปิดเผยข้อมูลหน่วยงานภาครัฐ (Agency Metadata) ผ่าน Resource `agencies://list` และ Tool `list_agency` รองรับทั้ง SSE Transport และ Streamable-HTTP Transport

## 2.5 Agent-to-Agent Protocol (A2A)

A2A (Agent-to-Agent) เป็นโปรโตคอลที่พัฒนาโดย Google สำหรับการสื่อสารระหว่าง AI Agent ต่าง ๆ ที่อาจพัฒนาด้วย Framework คนละตัว A2A เน้นการค้นพบ (Discovery) ว่ามี Agent ใดบ้างที่พร้อมให้บริการ และการแลกเปลี่ยนข้อมูลระหว่าง Agent อย่างเป็นมาตรฐาน

ในโครงงานนี้ รองรับ A2A เป็นหนึ่งใน Connection Type ของหน่วยงาน เพื่อให้ระบบมีความยืดหยุ่นในอนาคตเมื่อหน่วยงานต่าง ๆ พัฒนา AI Agent เป็นของตนเอง

## 2.6 RESTful API Architecture

REST (Representational State Transfer) เป็นรูปแบบสถาปัตยกรรมสำหรับการออกแบบ API ที่ใช้ HTTP Method (GET, POST, PUT, DELETE) ในการจัดการทรัพยากร (Resources) โดยมีหลักการสำคัญ ได้แก่ Client-Server, Stateless, Cacheable และ Uniform Interface

ในโครงงานนี้ Backend API ถูกออกแบบตามหลัก RESTful โดยใช้ FastAPI ซึ่งเป็น Python Web Framework ที่รองรับ Async/Await อย่างเต็มรูปแบบ มีการสร้าง API Documentation อัตโนมัติผ่าน OpenAPI Specification (Swagger UI ที่ `/docs`)

## 2.7 Single Page Application (SPA) และ React

Single Page Application (SPA) คือ Web Application ที่โหลดหน้า HTML เพียงครั้งเดียว จากนั้นอัปเดตเนื้อหาโดยใช้ JavaScript โดยไม่ต้อง Reload หน้าทั้งหมด ทำให้ผู้ใช้ได้รับประสบการณ์ที่ลื่นไหลคล้ายแอปพลิเคชัน Native

**React** เป็น JavaScript Library สำหรับสร้าง User Interface พัฒนาโดย Meta (Facebook) ใช้แนวคิด Component-based Architecture ที่แบ่ง UI ออกเป็นส่วนย่อย ๆ ที่นำกลับมาใช้ซ้ำได้ React ใช้ Virtual DOM ในการอัปเดตเฉพาะส่วนที่เปลี่ยนแปลง ทำให้มีประสิทธิภาพสูง

ในโครงงานนี้ใช้ React ร่วมกับ TypeScript เพื่อเพิ่มความปลอดภัยด้าน Type และ Vite เป็น Build Tool สำหรับความเร็วในการพัฒนา

## 2.8 JWT Authentication

JSON Web Token (JWT) เป็นมาตรฐาน (RFC 7519) สำหรับการส่งข้อมูลอย่างปลอดภัยระหว่างสองฝ่ายในรูปแบบ JSON Object ที่ถูกเข้ารหัส JWT ประกอบด้วย 3 ส่วน: Header (อัลกอริทึมที่ใช้), Payload (ข้อมูล), Signature (ลายเซ็นดิจิทัล)

ในโครงงานนี้ ใช้ JWT ร่วมกับ HS256 Algorithm สำหรับระบบ Authentication ของ Admin Portal โดยใช้ python-jose Library สำหรับสร้างและตรวจสอบ Token และ bcrypt สำหรับ Hash รหัสผ่าน Token มีอายุ 10,080 นาที (7 วัน) ก่อนหมดอายุ

## 2.9 Docker และ Container Technology

Docker เป็นแพลตฟอร์มสำหรับพัฒนา จัดส่ง และรันแอปพลิเคชันภายใน Container ซึ่ง Container คือสภาพแวดล้อมที่แยกตัว (Isolated) แต่ใช้ Kernel ของ Host OS ร่วมกัน ทำให้เบากว่า Virtual Machine

Docker Compose ใช้สำหรับจัดการหลาย Container พร้อมกัน ด้วยไฟล์ YAML ที่กำหนดค่า Services, Networks และ Volumes

ในโครงงานนี้ใช้ Docker Compose จัดการ 3 Services ได้แก่ PostgreSQL (ฐานข้อมูล), Backend (FastAPI), Nginx (Reverse Proxy + Static Files)

## 2.10 งานวิจัยและโครงงานที่เกี่ยวข้อง

### 2.10.1 ระบบ Chatbot ภาครัฐที่มีอยู่

ปัจจุบันหน่วยงานภาครัฐหลายแห่งได้พัฒนาระบบ Chatbot สำหรับให้บริการข้อมูลแก่ประชาชน เช่น Chatbot ของสำนักงานประกันสังคม และ Chatbot ของกรมสรรพากร อย่างไรก็ตาม Chatbot เหล่านี้ทำงานแยกส่วนกัน ประชาชนต้องทราบก่อนว่าคำถามของตนเกี่ยวข้องกับหน่วยงานใด จึงจะไปใช้ Chatbot ของหน่วยงานนั้น ๆ ได้ ซึ่งเป็นข้อจำกัดที่โครงงานนี้ต้องการแก้ไข

### 2.10.2 Multi-Agent LLM Systems

งานวิจัยเรื่อง "AutoGen: Enabling Next-Gen LLM Applications via Multi-Agent Conversation" โดย Wu et al. (2023) นำเสนอ Framework สำหรับสร้างระบบ Multi-Agent ที่ Agent สามารถสนทนาและทำงานร่วมกันได้ งานวิจัยนี้แสดงให้เห็นว่าการใช้หลาย Agent ร่วมกันสามารถแก้ปัญหาที่ซับซ้อนได้ดีกว่าการใช้ Agent ตัวเดียว

### 2.10.3 Retrieval-Augmented Generation (RAG)

งานวิจัยเรื่อง "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks" โดย Lewis et al. (2020) นำเสนอแนวคิด RAG ที่ผสมผสานการดึงข้อมูลจากแหล่งข้อมูลภายนอกเข้ากับความสามารถในการสร้างข้อความของ LLM ทำให้คำตอบมีความถูกต้องและอ้างอิงได้ โครงงานนี้ใช้แนวคิดคล้ายคลึงกันในการดึงข้อมูลจากหน่วยงานภาครัฐมาประกอบการสร้างคำตอบ

### 2.10.4 เปรียบเทียบงานที่เกี่ยวข้อง

| หัวข้อเปรียบเทียบ | Chatbot ภาครัฐทั่วไป | ระบบ RAG ทั่วไป | โครงงานนี้ |
|---|---|---|---|
| จำนวนแหล่งข้อมูล | 1 หน่วยงาน | 1 ฐานข้อมูล | หลายหน่วยงานพร้อมกัน |
| การ Route คำถาม | ไม่มี | ไม่มี | Keyword + LangGraph |
| Protocol | HTTP API | Vector DB | API + MCP + A2A |
| การสังเคราะห์คำตอบ | Rule-based | LLM | LLM + Multi-source |
| การจัดการหน่วยงาน | Hard-coded | ไม่มี | Dynamic CRUD |
| Dashboard | จำกัด | ไม่มี | มี Analytics |
