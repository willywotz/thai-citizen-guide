# OneChat API — v4: Streaming (SSE)

← [ภาพรวม API ทั้งหมด](../srs/03-api.md)

---

## 1. สรุป

`/v4/chat` ให้ข้อมูลชุดเดียวกับ [`/v3/chat`](v3.md) เป๊ะ แต่เปลี่ยนรูปแบบการรับส่งเป็น **SSE (Server-Sent Events)** — เปิด connection ค้างไว้แล้วส่ง event ทีละขั้นแบบ real-time แทนการรอจนเสร็จแล้วส่ง JSON ก้อนเดียว

**ทำไมต้องมี v4:** pipeline ใช้เวลานานได้ถึง ~30–90 วินาที (หน่วยงานปลายทางเป็น LLM agent ที่ตอบช้า) ถ้ารอแบบ v3 หน้าเว็บจะค้างเงียบทั้งช่วงนั้น v4 ส่งสัญญาณบอกทุกขั้นตอนเพื่อให้แสดง progress ได้ ("กำลังถามกรม X...")

**ใช้เมื่อ:** ต้องการแสดงความคืบหน้าระหว่างรอคำตอบ (progress bar, ทีละหน่วยงาน)

| | รายละเอียด |
|---|---|
| Response type | SSE stream (`text/event-stream`) |
| ข้อมูลใน event สุดท้าย | เหมือน `ChatV3Data` ของ v3 ทุกประการ |
| LLM synthesis | ✗ ไม่มี — เหมือน v3 |
| Structured sections | ✓ มี (ใน event `answer`) |
| Debug struct | ✓ มี (ใน event `answer`) |
| Smart fallback (ตอบไม่ได้) | ✓ มี — เหมือน v3 ทุกประการ |
| Inline citation | ✗ ไม่มี (ดู [v5](v5.md)) |

---

## 2. Endpoint

```
POST /v4/chat
Content-Type: application/json
Accept: text/event-stream
```

Response headers:

```
Content-Type: text/event-stream
Cache-Control: no-cache
X-Accel-Buffering: no
```

> **ห้ามใช้ `EventSource` ของ browser** — `EventSource` รองรับเฉพาะ `GET` แต่ v4 เป็น `POST` ที่มี body จึงต้องใช้ `fetch()` แล้วอ่าน `ReadableStream` ของ response body เอง

---

## 3. Request Body

ใช้ schema เดียวกันทุกเวอร์ชัน (v1–v5)

```json
{
  "query": "อยากเปิดร้านอาหารและขายเหล้า ต้องขออนุญาตอะไรบ้าง",
  "mcp_endpoint_url": "http://185.84.161.145/mcp",
  "session_id": "abc-123"
}
```

| Field | Type | จำเป็น | คำอธิบาย |
|---|---|---|---|
| `query` | string | ✅ | คำถามจากผู้ใช้; whitespace ซ้ำถูก normalize; ว่างล้วน → `400` |
| `mcp_endpoint_url` | string (URL) | ✅ | MCP catalog endpoint สำหรับ discover หน่วยงานแบบ runtime; ถูก SSRF guard (บล็อก IP วงใน) |
| `session_id` | string | ❌ | สำหรับ multi-turn; ไม่ส่ง → server สร้าง UUID ใหม่ (ไม่มี history) |

ส่ง field เกิน → `400` (`extra = "forbid"`)

## 4. Response — SSE Event Stream

### 4.1 รูปแบบ SSE

ทุก event ส่งมาเป็นบล็อกข้อความ ปิดท้ายด้วยบรรทัดว่าง:

```
event: <ชื่อ event>
data: <JSON หนึ่งบรรทัด>

```

ตัวอย่าง raw จริงจาก wire:

```
event: step
data: {"name": "discover", "status": "running", "ms": null}

event: agencies
data: {"agencies": [...], "count": 3}

```

- `data:` เป็น JSON บรรทัดเดียวเสมอ — parse ด้วย `JSON.parse()` ได้ตรงๆ
- ข้อความภาษาไทยส่งเป็น UTF-8 ไม่ escape (`ensure_ascii=false`)
- event แต่ละตัวคั่นด้วย `\n\n`

### 4.2 ลำดับ event (กรณี `intent = search`)

```
step      {name: "discover",   status: "running"}      <- เปิด connection ปุ๊บ ส่งทันที
step      {name: "discover",   status: "done"}
agencies  {agencies: [...], count: N}
step      {name: "classify",   status: "running"}
step      {name: "classify",   status: "done"}
intent    {intent: "search", ...}
routing   {sub_questions: [...]}
step      {name: "invoke",     status: "running"}
agency_start      {agency_id: "A", ...}                  <- ส่งทีเดียวทุกหน่วยงานที่จะเรียก
agency_start      {agency_id: "B", ...}
agency_responded  {agency_id: "B", status: "ok"}         <- ทยอยมาตามที่หน่วยงานตอบกลับ (ไม่ตามลำดับ)
agency_responded  {agency_id: "A", status: "error"}
step      {name: "invoke",     status: "done"}
step      {name: "verify",     status: "running"}
agency_verified   {agency_id: "B", status: "passed", relevance_score: 1.0}
step      {name: "verify",     status: "done"}
step      {name: "synthesize", status: "running"}
step      {name: "synthesize", status: "done"}
answer    {answer: "...", sections: [...], errors: [...], debug: {...}}
done      {session_id: "...", total_ms: 66351}
```

กรณี `chitchat` / `capability` / `out_of_scope` ข้ามขั้น `invoke`/`verify` ทั้งหมด (จาก `classify` ไป `synthesize` ตรง)

หลักการสำคัญ:
- ทุก step มาเป็นคู่ `running` แล้วตามด้วย `done` เสมอ
- `done` คือ event ปิดท้ายเสมอ หลังจากนั้น server ปิด connection
- ถ้า connection ปิดโดยไม่มี `done` ถือว่าเกิดข้อผิดพลาด (ดูข้อ 6)

### 4.3 Event Reference

#### `step`

```json
{ "name": "classify", "status": "running", "ms": null }
{ "name": "classify", "status": "done",    "ms": 5580.5 }
```

| name | ทำอะไร | เวลาโดยทั่วไป |
|---|---|---|
| `discover` | ดึงรายชื่อหน่วยงานจาก MCP catalog | ~0.5s (ถ้า endpoint ล่ม รอนานถึง ~30s) |
| `classify` | LLM วิเคราะห์คำถาม จัดประเภท intent + แตกคำถามย่อย | ~5–12s |
| `invoke` | ยิงคำถามไปทุกหน่วยงานพร้อมกัน | สูงสุด ~60s (ตาม timeout ของหน่วยงานช้าสุด) |
| `verify` | LLM ให้คะแนนความเกี่ยวข้องของแต่ละคำตอบ | ~1–3s |
| `synthesize` | ประกอบคำตอบสุดท้าย (raw concat, ไม่มี LLM) | แปรผัน |

#### `agencies` — รายชื่อหน่วยงานที่ค้นพบ

```json
{ "agencies": [{ "id": "...", "name": "กรมการปกครอง", "description": "...", "data_scope": ["..."] }], "count": 3, "errors": [] }
```

`count` อาจเป็น `0` ถ้า MCP catalog ล่ม — `errors[]` มีค่าเฉพาะตอนนั้น (`errorType: "MCPDiscoveryError"`)

#### `intent` — ผลการจัดประเภทคำถาม

```json
{ "intent": "search", "normalized_query": "...", "reasoning": "..." }
```

| Field | คำอธิบาย |
|---|---|
| `intent` | `search` / `chitchat` / `capability` / `out_of_scope` |
| `normalized_query` | คำถามที่ปรับให้สมบูรณ์แล้ว (เติมบริบทจากประวัติ, ปรับภาษา) |

#### `routing` — การแตกคำถามย่อยและจับคู่หน่วยงาน

ส่งเฉพาะตอน `intent = search`

```json
{ "sub_questions": [{ "section_label": "...", "broadcast": false, "agencies": [{ "id": "...", "name": "...", "query": "..." }] }] }
```

`broadcast: true` = confidence ต่ำกว่า 0.9 → ถามทุกหน่วยงาน; `false` = เจาะจงหน่วยงานที่ LLM มั่นใจ

#### `agency_start` / `agency_responded` / `agency_verified`

วงจรชีวิตของหน่วยงานหนึ่ง:

```
สำเร็จ:  agency_start -> agency_responded(ok) -> agency_verified(passed | rejected)
ล้มเหลว: agency_start -> agency_responded(error)
```

| event | ส่งเมื่อ | field สำคัญ |
|---|---|---|
| `agency_start` | เริ่มขั้น invoke (ส่งทีเดียวทุกหน่วยงาน) | `agency_id`, `agency_name`, `query`, `section_label` |
| `agency_responded` | หน่วยงานตอบกลับหรือ fail (ทยอยมา ไม่ตามลำดับ) | `status`: `ok`/`error`, `error_type` |
| `agency_verified` | ผลตรวจ relevance (เฉพาะที่ `responded.status = ok`) | `status`: `passed`/`rejected`, `relevance_score` |

หน่วยงานที่ `agency_responded.status = "error"` จะไม่มี `agency_verified` ตามมา

#### `answer` — คำตอบสุดท้าย

ข้อมูลชุดเดียวกับ `data` ของ [`/v3/chat`](v3.md#4-response-200) — ดูโครงสร้างเต็ม รวมพฤติกรรม smart fallback ที่ [v3 § 5](v3.md#5-พฤติกรรมตาม-intent--กรณีพิเศษ) และ `debug` struct เต็มที่ [v3 § 6](v3.md#6-datadebug)

```json
{
  "answer": "## โอนที่ดิน\n\n**กรมที่ดิน**\n\nสวัสดีค่ะ สำหรับขั้นตอน...",
  "sections": [{ "title": "โอนที่ดิน", "agencies": [{ "id": "...", "name": "กรมที่ดิน", "query": "...", "content": "..." }] }],
  "errors": [{ "agency": "...", "name": "กรมการปกครอง", "errorType": "TimeoutError", "message": "agency timed out after 60000 ms" }],
  "debug": { "...": "ดู v3 § 6" }
}
```

`sections[]` เรียง**ตามลำดับ sub-question ที่ user ถาม** เสมอ — ไม่ใช่ตามลำดับที่แต่ละหน่วยงานตอบเสร็จ (การตอบมาไม่พร้อมกันของ `agency_responded` ไม่กระทบลำดับ section สุดท้าย)

#### `done` — จบ stream

```json
{ "session_id": "f4ea61cc-47ac-4e46-bd6c-9cadc54ba999", "total_ms": 66351 }
```

event สุดท้ายเสมอ หลังจากนี้ server ปิด connection

#### `error` — ข้อผิดพลาดร้ายแรง (reserved)

ในทางปฏิบัติแทบไม่ถูกส่ง — ทุกขั้นตอนดักจับ error ของตัวเองไว้หมดแล้ว (MCP ล่ม → `agencies.count = 0`, หน่วยงาน error → `agency_responded.status = "error"` + `answer.errors`) ถ้าเกิดข้อผิดพลาดที่ดักไม่ได้จริงๆ connection จะถูกตัดโดยไม่มี `done` — **ใช้เกณฑ์ "ได้ `done` หรือยัง" เป็นสัญญาณความสำเร็จหลัก ไม่ใช่รอ event `error`**

## 6. การจัดการข้อผิดพลาด (เฉพาะ stream)

| สถานการณ์ | สิ่งที่เกิดขึ้น | Frontend ควรทำ |
|---|---|---|
| Request ผิด (ขาด/เกิน field) | HTTP `400` พร้อม JSON ก่อนเปิด stream | ตรวจ HTTP status ก่อนอ่าน stream |
| MCP catalog ล่ม | stream ทำงานปกติ แต่ `agencies.count = 0` และ `answer` เป็น smart fallback | แสดงข้อความว่าไม่พบหน่วยงาน |
| หน่วยงานหนึ่ง timeout/error | `agency_responded.status = "error"` + อยู่ใน `answer.errors` (หน่วยงานอื่นทำงานต่อ) | แสดง error เฉพาะหน่วยงานนั้น |
| Server error ที่ดักไม่ได้ | connection ถูกตัด ไม่มี `done` | เช็ค flag "ได้ `done` หรือยัง" |

**กฎทอง:** ความสำเร็จของ request วัดจากการได้รับ event `done` ไม่ใช่แค่ stream ปิด

## 8. ตัวอย่าง

```bash
curl -N -X POST http://<host>:8000/v4/chat \
  -H "Content-Type: application/json" \
  -d '{"query":"เปิดร้านอาหารต้องขออนุญาตอะไร","mcp_endpoint_url":"http://185.84.161.145/mcp"}'
```

แนวทางจัดการ state ฝั่ง frontend (map ด้วย `agency_id`, เขียนทับไม่ append):

| ส่วนของ state | อัปเดตจาก event |
|---|---|
| ขั้นตอนปัจจุบัน/สถานะแต่ละขั้น | `step` |
| รายชื่อหน่วยงาน | `agencies` |
| intent / routing | `intent`, `routing` |
| สถานะรายหน่วยงาน | `agency_start` → `agency_responded` → `agency_verified` (key = `agency_id`) |
| คำตอบสุดท้าย | `answer` |
| จบหรือยัง | `done` |

หมายเหตุประสิทธิภาพ: ตั้ง client timeout ≥ 120 วินาทีหรือใช้ `AbortController` แทนการตั้ง timeout — pipeline ใช้เวลาทั่วไป ~30–90 วินาที
