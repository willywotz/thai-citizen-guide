# OneChat API — v5: Streaming + Executive Summary

← [ภาพรวม API ทั้งหมด](../srs/03-api.md)

---

## 1. สรุป

`/v5/chat` ทำงานเหมือน [`/v4/chat`](v4.md) ทุกประการ (SSE, ลำดับ event เดียวกัน, pipeline เดียวกัน, `sections[]` เนื้อหาเดียวกันทุกตัวอักษร) เพิ่มเติมแค่จุดเดียว: บทสรุปภาพรวม (executive summary) พร้อมเลขอ้างอิงไว้บนสุดของ `answer`

| | v4 | v5 |
|---|---|---|
| เนื้อหา `sections[]` | คำตอบดิบของ agency ทั้งหมด | **เหมือนกันทุกตัวอักษร** — v5 ไม่ตัด ไม่แก้ ไม่ใส่เลขอ้างอิงใน section เลย |
| ด้านบนสุดของ `answer` | ไม่มี | บทสรุปสั้นๆ ที่ LLM เขียนจากทุก agency ที่ผ่านการตรวจสอบทั้ง request พร้อมเลขอ้างอิง `[n]` กำกับทุกข้อเท็จจริงสำคัญ |
| การอ้างอิงที่มา | ไม่มี | `references[]` — ขอบเขตเฉพาะบทสรุปเท่านั้น (sections ด้านล่างไม่มีเลขอ้างอิง) |
| หลาย agency ในหัวข้อเดียวกัน | เรียงต่อกันตรงๆ ไม่มีคำเชื่อม | เหมือน v4 เป๊ะ — **ไม่มี** LLM แต่งประโยคเชื่อมอีกต่อไป |
| ชื่อการสนทนา | ไม่มี | `thread_name` ใน event `intent` และ `done` |

**ใช้เมื่อ:** ต้องการให้ผู้ใช้เห็นภาพรวมคำตอบเร็วๆ ด้านบนสุดพร้อมอ้างอิงที่ตรวจสอบย้อนกลับได้ โดยยังมีเนื้อหาดิบครบถ้วนจาก agency อยู่ด้านล่างเหมือน v4 ทุกประการ

| | รายละเอียด |
|---|---|
| Response type | SSE stream (`text/event-stream`) |
| LLM synthesis ของเนื้อหา agency | ✗ ไม่มี — `sections[]` เป็น raw content เหมือน v4 ทุกประการ |
| LLM สร้างบทสรุปด้านบน | ✓ มี — 1 ครั้งต่อ request (ไม่ใช่ต่อ agency) ดูข้อ 4.3 |
| Structured sections | ✓ มี (เหมือน v3/v4 เป๊ะ) |
| Debug struct | ✓ มี (เหมือน v3/v4) |
| Smart fallback (ตอบไม่ได้) | ✓ มี — เหมือน v3/v4 ทุกประการ (ไม่มีการตัด filler พิเศษอีกต่อไป) |
| Inline citation | ✓ มี — แต่อยู่ในบทสรุปเท่านั้น ไม่ปรากฏใน `sections[]` |

## 2. Endpoint

```
POST /v5/chat
Content-Type: application/json
Accept: text/event-stream
```

Request body, response headers, รูปแบบ SSE ดิบ — เหมือน v4 ทุกประการ ดู [v4 § 2–4.1](v4.md#2-endpoint)

## 4. Response — SSE Event Stream

### 4.1 ลำดับ event — ต่างจาก v4 ตรงไหน

```
...
step      {name: "invoke",     status: "done"}
step      {name: "verify",     status: "running"}
agency_verified   {agency_id: "B", status: "passed", relevance_score: 1.0}
step      {name: "verify",     status: "done"}
step      {name: "summarize",  status: "running"}   <- ★ ใหม่ ★ ไม่มีใน v4
step      {name: "summarize",  status: "done"}       <- ★ ใหม่ ★
step      {name: "synthesize", status: "running"}
step      {name: "synthesize", status: "done"}
answer    {answer: "...", summary: "...", sections: [...], references: [...], errors: [...], debug: {...}}
done      {session_id: "...", total_ms: 66351, thread_name: "ค่าธรรมเนียมโอนที่ดิน"}
```

ขั้น `summarize` อยู่ระหว่าง `verify` กับ `synthesize` เท่านั้น — ทุกขั้นตอนก่อนหน้านั้น (discover, classify, invoke, verify) เหมือน v4 เป๊ะ ดู step reference เต็มที่ [v4 § 4.3](v4.md#43-event-reference)

| name (ใหม่) | ทำอะไร | เวลาโดยทั่วไป |
|---|---|---|
| `summarize` | LLM เขียนบทสรุปภาพรวมจาก agency ที่ผ่านการตรวจสอบทั้งหมดของ request (1 ครั้งเดียว ไม่ใช่ต่อ agency) พร้อมเลขอ้างอิง | ~2–5s (LLM call เดียว ไม่ว่าจะมี agency กี่ราย) |

### 4.2 `intent` — เพิ่ม field `thread_name`

```json
{ "intent": "search", "normalized_query": "...", "reasoning": "...", "thread_name": "ค่าธรรมเนียมโอนที่ดิน" }
```

| Field | คำอธิบาย |
|---|---|
| `thread_name` | string \| null — ชื่อหัวข้อสนทนาสั้นๆ (3–8 คำ) สร้างโดย LLM — มีความหมายเฉพาะตอนเป็นคำถามแรกของ session (ดูข้อ 4.5) |

### 4.3 `answer` — เพิ่ม field `summary` + `references`

```json
{
  "answer": "สรุปครับ ค่าธรรมเนียมการโอนที่ดินอยู่ที่ 2% ของราคาประเมิน [1]\n\n**อ้างอิง**\n\n[1] กรมที่ดิน\n\n---\n\n## ค่าธรรมเนียมโอนที่ดิน\n\nค่าธรรมเนียมการโอนคือ 2% ของราคาประเมิน",
  "summary": "สรุปครับ ค่าธรรมเนียมการโอนที่ดินอยู่ที่ 2% ของราคาประเมิน [1]",
  "sections": [{ "title": "ค่าธรรมเนียมโอนที่ดิน", "agencies": [{ "id": "land", "name": "กรมที่ดิน", "query": "...", "content": "ค่าธรรมเนียมการโอนคือ 2% ของราคาประเมิน" }] }],
  "references": [{ "number": 1, "agency_id": "land", "agency_name": "กรมที่ดิน", "url": null }],
  "errors": [],
  "debug": { "...": "เหมือน v4/v3" }
}
```

| Field | คำอธิบาย |
|---|---|
| `summary` | string — บทสรุปภาพรวมอย่างเดียว (ไม่รวม section ด้านล่าง) เขียนโดย LLM จาก agency ที่ผ่านการตรวจสอบทั้ง request; ว่างเปล่า (`""`) ถ้าไม่มี agency ไหนผ่านเลย หรือถ้า LLM สร้างบทสรุปล้มเหลว (ดูพฤติกรรม degrade ด้านล่าง) |
| `references[]` | รายการอ้างอิงของ `summary` เท่านั้น — เรียงตามเลขที่ใช้ครั้งแรก หนึ่งรายการต่อ agency (agency ที่ตอบมากกว่า 1 หัวข้อในคำขอเดียวกันได้เลขอ้างอิง**เดียว**ทุกที่ ไม่ใช่เลขใหม่ต่อหัวข้อ) |
| `references[].number` | เลขอ้างอิง ตรงกับ `[n]` ใน `summary` |
| `references[].url` | ปัจจุบันเป็น `null` เสมอ (บทสรุปไม่ได้ผูกกับ URL อ้างอิงของหน่วยงานแบบ v4/v5 เดิม) |

`answer` ประกอบจาก 3 ส่วนตามลำดับ: `summary` → รายการอ้างอิง (ถ้ามี citation) → `---` → เนื้อหา section แบบเดียวกับ v4 (`## หัวข้อ` ตามด้วยเนื้อหาดิบของแต่ละ agency) `sections[].agencies[].content` เป็น raw content ล้วนๆ **ไม่มี** `[n]` ปนอยู่เลย ต่างจาก v5 เดิม

**เมื่อการสร้างบทสรุปล้มเหลว (LLM error หรือ output ผิดรูปแบบ):** ระบบ degrade แบบเงียบ — `summary` และ `references` เป็นค่าว่าง และ `answer` จะเหมือนกับที่ `/v4/chat` ส่งกลับสำหรับคำถามเดียวกันทุกประการ (ไม่มีข้อความแจ้งเตือนใดๆ แทรกเข้ามา)

### 4.4 Smart fallback — เหมือน v3/v4 ทุกประการ

พฤติกรรมเหมือน [v3 § 5](v3.md#5-พฤติกรรมตาม-intent--กรณีพิเศษ) ทุกประการ ไม่มีข้อยกเว้นเฉพาะ v5 อีกต่อไป (เดิม v5 เคยตัด filler เพิ่มในเคส agency เดียวตอบไม่ตรง — พฤติกรรมนี้ถูกลบออกพร้อมกับการลบกลไก trim ทั้งหมด)

- section ที่เกิดจาก fallback **ไม่เคยมีเลขอ้างอิง `[n]`** และไม่ปรากฏใน `references[]` เพราะไม่ใช่เนื้อหาที่ผ่านการตรวจ relevance จริง — `errors[]` ยังคงบันทึกหน่วยงานที่ reject ไว้เหมือนเดิม
- ถ้าไม่มี agency ไหนผ่านการตรวจสอบเลยในทั้ง request `summarize` ไม่ทำงาน (ไม่มี LLM call) — `summary`/`references` เป็นค่าว่าง `answer` เป็นข้อความ fallback เดียวกับ v3/v4

### 4.5 `done` — เพิ่ม field `thread_name`

```json
{ "session_id": "f4ea61cc-...", "total_ms": 66351, "thread_name": "ค่าธรรมเนียมโอนที่ดิน" }
```

**พฤติกรรมข้าม turn:** อ่าน `thread_name` จาก event `done` เท่านั้น (ไม่ใช่จาก `intent`):
- **คำถามแรกของ session**: `thread_name` ที่สร้างสดจาก LLM ถูกบันทึกและส่งกลับ
- **คำถามถัดไปในเซสชันเดียวกัน**: ค่าที่สร้างใหม่ใน event `intent` ถูก**ละเว้น** — backend ส่งชื่อเดิมที่บันทึกไว้ตอนคำถามแรกเสมอ กันชื่อ thread เปลี่ยนกลางบทสนทนา
- **ไม่มี `REDIS_URL`**: ไม่มี session persistence เลย → `thread_name` เป็น `null` เสมอทุกคำถาม

## 7. ตัวอย่าง

```bash
curl -N -X POST http://<host>:8000/v5/chat \
  -H "Content-Type: application/json" \
  -d '{"query":"ค่าธรรมเนียมโอนที่ดินเท่าไหร่","mcp_endpoint_url":"http://185.84.161.145/mcp"}'
```

หมายเหตุประสิทธิภาพ: ขั้น `summarize` เป็น LLM call เดียวต่อทั้ง request (ไม่ใช่ต่อ agency แบบกลไก trim เดิม) — เร็วกว่า v5 เดิมสุทธิเมื่อมีหลาย agency ในคำขอเดียว ดูรายละเอียดทั่วไปที่ [v4 § 8](v4.md#8-ตัวอย่าง)
